# Архитектура онлайн-записи в МИС

Как AI-ассистент записывает клиента в реальную медицинскую информационную систему (МИС) клиники — не ломая существующие процессы и не завися от конкретного вендора.

## Принцип: единый бэкенд-прокси

Ни сайт, ни бот **не обращаются к МИС напрямую**. Они ходят только в наш бэкенд (`/api/booking/*`), а тот уже общается с МИС через абстракцию транспорта.

```
Сайт ─┐
      ├─▶  Наш backend (/api/booking)  ─▶  BookingTransport  ─▶  МИС
Бот ──┘         кэш · идемпотентность ·        Local │ Medflex │ 1С
                антидабл · 152-ФЗ
```

Что это даёт:

- **Единая точка смены транспорта** — режим записи (ручной → МедФлекс → прямой в 1С) меняется под капотом, прозрачно для UI и бота.
- **Безопасность** — креды МИС/токены живут только на сервере (env), не во фронте.
- **Кэш расписания** — снижаем нагрузку на МИС и обходим её таймауты операций.
- **Идемпотентность и антидабл** — блокировка по слоту на нашей стороне, МИС — финальный арбитр.
- **152-ФЗ** — персональные данные проходят через наш контролируемый периметр.

## Три режима записи

| Режим | Суть | Когда |
|---|---|---|
| **Local** | Запись в нашу БД, регистратор переносит в МИС | MVP / клиника не готова открывать МИС |
| **Medflex** | Через партнёрский API МедФлекса (`Authorization: Token …`) | МедФлекс уже подключён к МИС клиники |
| **Direct (1С:БИТ УМЦ)** | Прямой HTTP-сервис расширения 1С через защищённый канал | Полная автономия, 0 ₽ за запись |

Все три реализуют один интерфейс `BookingTransport` (`getSlots`, `createBooking`) — переключение через переменную окружения `BOOKING_TRANSPORT`, без изменений в коде UI/бота.

## Почему так, а не «бот напрямую в МИС»

1. **МИС у каждой клиники своя** (1С:БИТ УМЦ, посредники вроде МедФлекса, агрегаторы). Прямая привязка к одному вендору = переписывать бота под каждого клиента.
2. **Слоты и занятость считает МИС** (по графику врача и длительности приёма) — нам не нужно строить календарь-движок, но нужно аккуратно кэшировать и не упереться в таймаут операции (обычно ≤ 10 c).
3. **Поиск/создание карты пациента** — на стороне МИС (по ФИО + дате рождения + телефону), защита от дублей пациентов там же.

## Ключевые ограничения и как они решаются

| Риск | Решение |
|---|---|
| Таймаут операции МИС (≤ 10 c) | Кэш расписания, ретраи, асинхронное подтверждение |
| Конкурентные слоты → двойная бронь | Lock по слоту на нашей стороне + МИС как финальный арбитр; статус «подтверждается» |
| МИС недоступна из интернета | Защищённый канал (VPN/туннель) или IP-allowlist + HTTPS; отдельный сервисный пользователь с минимальной ролью |
| Медданные / 152-ФЗ / врачебная тайна | В записи-заявке собираем только контакт + желаемое время; диагноз/жалобы не храним как медданные; шифрование канала к МИС |

## Эволюционный путь

Платформа стартует на `Local` (работает сразу, 0 ₽ за запись), параллельно подключается `Medflex` как быстрый real-time, а к `Direct`-записи в 1С приходим, когда клиника готова открыть МИС. **Для фронтенда и бота на всех этапах ничего не меняется** — вся разница инкапсулирована в транспорте.
