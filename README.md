# AI Assistant Platform

**Multi-tenant платформа AI-ассистентов для бизнеса.** Один движок обслуживает множество клиентов: каждый ассистент настраивается без выката кода, встраивается на любой сайт одной строкой, ведёт диалог на Claude, захватывает заявки в Telegram и умеет записывать клиентов в их реальные системы (МИС/CRM).

> 📌 Это **showcase-репозиторий** (архитектура + ключевые решения + избранные фрагменты кода). Боевой код — приватный, т.к. обслуживает реальных клиентов. Живые внедрения: AI-ассистент «София» на сайте бронирования апартаментов, ассистенты в клиниках и строительной компании.

## Проблема

Малый и средний бизнес теряет обращения: менеджеры не отвечают ночью и в выходные, а на однотипные вопросы («сколько стоит», «есть ли время», «как записаться») уходит всё их время. Готовые чат-виджеты (Jivo и аналоги) — это просто труба для переписки: за ней всё равно должен сидеть живой человек.

## Решение

Платформа, на которой за минуты разворачивается AI-ассистент под конкретный бизнес:

- **отвечает 24/7** на естественном языке (Claude), зная цены/услуги/часы из живого источника;
- **доводит до заявки** и забирает контакт;
- **присылает лид владельцу в Telegram** вместе с историей диалога;
- **записывает клиента** прямо в его систему (для клиник — онлайн-запись в МИС);
- всё это — **multi-tenant**: новый клиент = строка в БД, без релиза.

## Архитектура

```
                        ┌──────────────────────────────────────────────┐
   Сайт клиента  ──────▶│  widget.js  (одна строка <script>, CORS)      │
   (любой домен)        └───────────────┬──────────────────────────────┘
                                        │
                         ┌──────────────▼───────────────────────────────┐
                         │            ПЛАТФОРМА (Next.js)                 │
                         │                                               │
   Telegram  ◀───────────┤  /api/bots/[slug]/chat   — стриминг Claude    │
   (лиды владельцу)      │     ├─ system_prompt из БД (per-tenant)       │
                         │     ├─ live-контекст по context_url           │
                         │     ├─ актуальные дата/время (МСК) в коде      │
                         │     └─ парсинг ссылок из сообщения             │
                         │  /api/bots/[slug]/lead   — захват контакта     │
                         │  /api/booking/*          — запись на приём     │
                         │  /aiadmin, /{tenant}     — админ-дашборды      │
                         └──────────────┬───────────────┬────────────────┘
                                        │               │
                              ┌─────────▼──────┐  ┌─────▼─────────────────┐
                              │  SQLite         │  │  Booking transport     │
                              │  clients,       │  │  Local │ Medflex │ 1С  │
                              │  leads,         │  │  (абстракция МИС/CRM)  │
                              │  messages, ...  │  └───────────────────────┘
                              └────────────────┘
```

### Ключевые архитектурные решения

**1. Multi-tenant через конфигурацию, а не через код.**
Каждый клиент — строка в таблице `clients`: `slug`, `system_prompt`, `model`, `context_url`, `tg_token`, настройки виджета, `rate_limit`. Завести нового ассистента = добавить строку из админки. Ноль деплоев на онбординг клиента.

**2. Живой контекст вместо «вшитых» знаний.**
Ассистент не «помнит» цены в промпте (они устаревают), а подтягивает их в рантайме по `context_url` клиента. Плюс актуальные дата/время считаются **в коде по МСК** и подмешиваются в промпт — модель не путается в «завтра» и «работаете в выходные».

**3. Захват лида устойчив к «молчунам».**
Если посетитель написал несколько раз, но контакт не оставил — на N-м сообщении владельцу всё равно уходит уведомление с куском диалога. Обычные виджеты так не умеют: там либо оператор онлайн, либо тишина.

**4. Запись в чужие системы — через абстракцию транспорта.**
Интерфейс `BookingTransport` с реализациями `Local` / `Medflex` / `Direct (1С:БИТ УМЦ)`. Фронт и бот всегда ходят только в наш бэкенд; транспорт к реальной МИС меняется под капотом прозрачно (от ручного режима до real-time-записи) без изменений в UI. См. [docs/booking-architecture.md](docs/booking-architecture.md).

**5. Виджет — изолированный, одной строкой.**
`<script src="…/widget.js" data-bot="slug">` на любой сайт. CORS на внешние домены, изоляция стилей, ленивая загрузка.

## Self-serve онбординг («нанять AI-ассистента за 10 минут»)

Поверх платформы построен self-serve-поток для вертикали клиник — чтобы клиент завёл ассистента сам, без менеджера:

```
Диалог-«архитектор»  →  Провижининг  →  Дашборд
(ArchitectBot)          (авто-создание)   (лиды + аналитика)
```

- **Архитектор, а не конструктор.** Клиент не пишет ТЗ на бота — вместо этого AI-бот сам расспрашивает (роль, специальность, сайт, Telegram) и собирает конфиг. Решение проблемы «пользователь не знает, что ему нужно».
- **Провижининг.** По собранному конфигу автоматически заводится арендатор (строка в `clients`), генерируется system prompt, подключается виджет и Telegram — клиент сразу получает рабочего ассистента.
- **Дашборд.** Лиды со статусами и аналитика обращений.

> Реализовано как интерактивное демо (вертикаль «AI-администратор для клиник»). Биллинг и публичная регистрация — в roadmap.

## Возможности

| Область | Что умеет |
|---|---|
| Диалог | Стриминг Claude, per-tenant system prompt, живой контекст, парсинг ссылок |
| Лиды | Захват контакта → SQLite + Telegram владельцу с историей; «ловля» незавершённых диалогов |
| Запись | Онлайн-запись в МИС (МедФлекс / 1С), нормализация телефона, защита от двойной брони |
| Админка | Лиды со статусами, просмотр диалогов, календарь записи, услуги, статистика |
| Multi-tenant | Новый ассистент из админки без релиза; настройка модели/промпта/виджета per-client |
| Стоимость | По умолчанию экономичная модель (Haiku), переключается per-tenant |

## Технологии

`Next.js 16 (App Router)` · `React 19` · `TypeScript (strict)` · `Tailwind CSS 4` · `better-sqlite3` · `@anthropic-ai/sdk (Claude)` · `Telegram Bot API` · интеграции: `Yandex Direct`, `МедФлекс API`, `1С:БИТ УМЦ`

## Избранные фрагменты кода

- [Chat engine — потоковый ответ ассистента](snippets/chat-engine.ts) — per-tenant промпт, pre-calc дат в коде (не доверяем календарь модели), живой `context_url`, SSRF-safe фетч ссылок, стриминг.

## Ограничения / зона роста

- Хранилище — SQLite (подходит под текущий масштаб; для роста — переход на Postgres + изоляция арендаторов).
- RAG/векторный поиск не используется: ставка на живой `context_url`, что точнее для динамичных данных (цены/слоты), но не покрывает Q&A по большим документам.
- Биллинг и self-serve-онбординг — в проработке (сейчас провижининг клиента — через админку).
